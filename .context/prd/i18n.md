# PRD: Internationalization (i18n)

> **Status**: Active · **Created**: 2026-07-08 · **Source**: doc/prd/i18n.md

## Purpose & scope

Provide internationalization infrastructure for the whole monorepo: multi-language text translation, localized number/currency/date formatting, and locale-aware routing. Delivered as a base Nuxt layer (`layers/i18n`, published as `@mmshark/i18n-layer`) that other layers and apps extend with their own namespaced translations.

**In scope**: base `@nuxtjs/i18n` configuration, common shared translations, number/date format definitions, locale detection and switching, the distributed layer-extension pattern, SSR-compatible locale handling.

**Out of scope**: domain-specific translation content (owned by each layer), CMS/database-driven translations, translation management UI, machine translation.

> **Reality check**: the infrastructure below exists and is correctly configured, but **nothing in the product consumes it today** — see [Current status](#current-status). This PRD specifies the target; it does not claim the target is met.

## Requirements

### Configuration (base layer)

Owned by [`layers/i18n/nuxt.config.ts`](../../layers/i18n/nuxt.config.ts):

| Requirement | Target |
|---|---|
| Locales | `en` (default, en-US) and `es` (es-ES); architecture must allow adding locales without touching consumers |
| URL strategy | `prefix_except_default` — `/dashboard` (en), `/es/dashboard` (es) |
| Loading | Lazy-load translation files per locale |
| Number formats | `currency` (USD for en, EUR for es), `decimal` (2 fraction digits), `percent` — per locale |
| Date/time formats | `short` and `long` named formats per locale; add `time` when a consumer needs it |
| Locale persistence | Browser-language detection on first visit, persisted via cookie, fallback to `en` |

### Distributed translation architecture

Each layer ships its own translation files under its own namespace and merges them via its `nuxt.config.ts` `i18n.locales[].file` entries; the base layer provides shared config and the `common.*` / `navigation.*` / `status.*` / `errors.*` / `validation.*` namespaces.

- Key naming: `{namespace}.{category?}.{key}`, camelCase, ≤3 nesting levels (e.g. `common.save`, `billing.actions.upgrade`, `auth.errors.invalidCredentials`).
- en/es key parity is mandatory for every translation file; missing keys fall back to the default locale.
- Extension mechanism in use today: [`layers/billing/nuxt.config.ts`](../../layers/billing/nuxt.config.ts) and [`apps/saas/nuxt.config.ts`](../../apps/saas/nuxt.config.ts) add `billing.json` / `app.json` per locale.

### Consumption (the actual product behavior)

| Requirement | Detail |
|---|---|
| Translated UI | All user-facing strings rendered through `$t()` / `useI18n().t` — no hardcoded copy in components, navigation config, toasts, or validation messages |
| Localized formatting | Currency/numbers through `$n()` with the shared formats; dates through `$d()`; no hardcoded `'en-US'` locale arguments |
| Language switcher | A visible control (user menu or settings) listing available locales, switching via `setLocale()`, persisting the choice |
| Locale-aware navigation | Internal links and programmatic redirects use `localePath()` / `switchLocalePath()` so the `/es` prefix survives navigation and middleware redirects |
| Server-side locale | API routes and server-rendered output resolve locale from cookie/`Accept-Language`; localized transactional email content is deferred to the email epic |

### Testing

- Key-parity check across locales for every translation file (unit).
- Formatting assertions per locale (currency separators, date patterns) (unit).
- Locale switching, cookie persistence, and prefixed-URL navigation (e2e).

## Open issues & risks

1. **Documentation drift (the reason this rewrite exists)**: `doc/analysis/gap-analysis-code-vs-prd.md` rates i18n "~100% Complete / A+"; `layers/i18n/README.md` still says "Nuxt 3" and shows a `useLocalization` composable that does not exist. Both must be corrected or retired — do not trust them.
2. **Prefixed routes are live but the code is locale-unaware**: with `prefix_except_default`, `/es/*` routes are generated today, yet middleware and links use hardcoded paths (`layers/auth/middleware/auth.ts:16`), so a Spanish visitor is silently bounced back to unprefixed English routes. Either adopt `localePath()` everywhere (E13) or the ES route tree remains a trap.
3. **Formatting divergence**: three different formatting approaches coexist (hardcoded `'en-US'`, `Intl.NumberFormat(undefined)`, and the unused i18n `numberFormats`). Consolidating on `$n()`/`$d()` is part of the adoption sweep.
4. **Cookie/detection behavior is unspecified**: relying on module defaults means locale persistence semantics can shift on a module upgrade; pin the `detectBrowserLanguage` config explicitly when wiring the switcher.
