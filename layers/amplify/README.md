# Amplify Layer

AWS Amplify Gen2 integration layer for Nuxt 4 applications (`@mmshark/amplify-layer`). Provides the client/server Amplify configuration, the GraphQL Data client, S3 storage helpers, SSR-safe server utilities for Nitro API routes, the invoke helper for the backend's privileged `workspace-membership` Lambda, and a small structured logger used across server code.

## Overview

- 🔗 **GraphQL Data Client** — type-safe operations against the AppSync API generated from `apps/backend/amplify/data/resource.ts`
- 🔐 **Authentication integration** — Cognito user pool sign-in/session, exposed via `$Amplify.Auth`
- 📁 **Storage access** — S3 upload/URL helpers via `$Amplify.Storage`
- 🖥️ **SSR support** — cookie-backed token/credentials providers so server-rendered pages and Nitro routes see the caller's session
- ⚡ **Server helpers** — `withAmplifyAuth`/`withAmplifyPublic` + preconfigured Data clients for Nitro `server/api/**` routes (which have no Nuxt app context)
- 🧩 **Workspace membership invoke helper** — shared by the `workspaces` and `billing` layers to call the backend's privileged `workspace-membership` Lambda
- 🪵 **Structured logger** — environment-aware `createLogger(scope)` used instead of ad hoc `console.*`

## Architecture

```
layers/amplify/
├── plugins/
│   ├── 01.amplify.client.ts       # Browser: Amplify.configure + $Amplify provide
│   └── 01.amplify.server.ts       # SSR/Nuxt context: cookie-backed Amplify.configure + $Amplify provide
├── server/
│   └── utils/
│       ├── amplify.ts             # withAmplifyAuth/withAmplifyPublic, Data client factories, getAwsCredentials
│       ├── workspaceGroups.ts     # Cognito group-name helpers for the group-per-workspace model
│       └── workspaceMembership.ts # invokeWorkspaceMembership() + token/invitation-token helpers
├── utils/
│   └── logger.ts                  # createLogger(scope) structured logger
├── types/
│   └── amplify.d.ts               # Declares the $Amplify Nuxt-app property shape
├── amplify_outputs.json           # Generated by `ampx generate outputs` (gitignored, not committed)
└── nuxt.config.ts
```

There is no `utils/graphql/` (no committed generated queries/mutations/subscriptions files) and no `utils/server.ts` "legacy" module — both were removed. All GraphQL access goes through the typed `models.*` API on the Data client (`client.models.<Model>.get/list/create/update/delete`), not raw `graphql()` documents, except where the code explicitly needs a custom operation.

Peer dependencies (`layers/amplify/package.json`): `nuxt`, `aws-amplify`, `@aws-amplify/core`, `@aws-amplify/auth`, `@aws-amplify/api`, `@aws-amplify/storage`. Direct dependency: `@aws-sdk/client-lambda` (used by `workspaceMembership.ts` to invoke the backend Lambda).

## Authorization Model

`apps/backend/amplify/data/resource.ts` sets `defaultAuthorizationMode: "userPool"`. The only `allow.publicApiKey()` grant in the whole schema is a **read-only** one on `SubscriptionPlan` (the public pricing page). There is **no general-purpose public API key** and **no IAM/identity-pool data client** — a previous `getServerIamDataClient()` existed briefly and was deliberately removed (see the code comment in `server/utils/amplify.ts`) because `allow.authenticated('identityPool')` authorized *any* signed-in user's raw Identity Pool credentials to hit AppSync directly, bypassing the routes' OWNER/ADMIN checks.

Tenant tables (`Workspace`, `WorkspaceMember`, `WorkspaceSubscription`, `WorkspaceInvitation`) are **read-only for clients**, gated by dynamic per-workspace Cognito groups (see [Workspace Group Helpers](#workspace-group-helpers-serverutilsworkspacegroupsts) below). Every write goes through one of three privileged backend Lambdas (`workspace-membership`, `stripe-webhook`, `post-confirmation`), each holding an `allow.resource(...)` grant — never through a client-authMode Data client. Server routes acting on behalf of a signed-in user use `getServerUserPoolDataClient()` inside `withAmplifyAuth`, so the caller's own `cognito:groups` claim governs what AppSync allows.

| Scenario | authMode | Helper |
|---|---|---|
| Signed-in user reading/writing data covered by their own owner/group claims | `userPool` | `withAmplifyAuth()` + `getServerUserPoolDataClient()` |
| Public `SubscriptionPlan` reads (landing page, pricing) | `apiKey` | `withAmplifyPublic()` + `getServerPublicDataClient()` |
| Tenant-table writes (workspace/member/invitation/subscription CRUD) | n/a — delegated | `invokeWorkspaceMembership()` (see below); the backend Lambda holds its own `allow.resource(...)` grant |
| Stripe webhook sync | n/a — separate Lambda | `apps/backend/amplify/functions/stripe-webhook/` (a public Function URL, not routed through this layer at all) |

## Client Plugin

`plugins/01.amplify.client.ts` runs `Amplify.configure(outputs, { ssr: true })` and creates a single `generateClient<Schema>({ config: outputs, authMode: 'userPool' })`, provided as `$Amplify.GraphQL.client`. It's a fixed-`authMode` client by design (fail-safe: you can't accidentally query with the wrong mode). For an operation that genuinely needs a different `authMode` client-side, construct one explicitly with `generateClient` rather than changing the shared instance.

**Provides:**
- `$Amplify.Auth` — the full `aws-amplify/auth` module (`signIn`, `signOut`, `fetchAuthSession`, `getCurrentUser`, ...)
- `$Amplify.GraphQL.client` — the userPool-authenticated Data client (`client.models.<Model>.*`, or `client.graphql(...)` for raw operations)
- `$Amplify.Storage` — `{ uploadData, getUrl }` from `aws-amplify/storage`

There is no `$Amplify.Data` namespace — that name appears only in older documentation/examples and does not exist in this layer's actual `$Amplify` shape (see `types/amplify.d.ts`).

## Server Plugin

`plugins/01.amplify.server.ts` mirrors the client plugin for Nuxt's SSR/app context (pages, components, middleware — anything with access to `useNuxtApp()`). It builds cookie-backed token/credentials providers from the incoming request and runs everything inside `runWithAmplifyServerContext`, providing the same `$Amplify.Auth` / `$Amplify.GraphQL.client` / `$Amplify.Storage` shape as the client plugin.

**This plugin does not help bare Nitro `server/api/**` routes** — those have no Nuxt app instance, so `useNuxtApp()` throws. Use the [Server Utilities](#server-utilities-serverutilsamplifyts) below instead.

## Server Utilities (`server/utils/amplify.ts`)

For Nitro API routes and Lambda-adjacent server code that don't have a Nuxt app context:

- **`withAmplifyAuth(event, callback)`** — runs `callback(contextSpec)` inside `runWithAmplifyServerContext`, with a token/credentials provider built from the request's cookies. Use for any route acting on behalf of a signed-in user.
- **`withAmplifyPublic(callback)`** — runs `callback(contextSpec)` with **no credentials provider at all** (sessionless). Only valid for `apiKey`-mode reads (`SubscriptionPlan`); it cannot reach any tenant data.
- **`getServerUserPoolDataClient()`** — `generateClient<Schema>({ config: amplifyConfig, authMode: 'userPool' })`. Use inside `withAmplifyAuth`.
- **`getServerPublicDataClient()`** — `generateClient<Schema>({ config: amplifyConfig, authMode: 'apiKey' })`. Use inside `withAmplifyPublic`, for `SubscriptionPlan` reads only.
- **`getAwsCredentials(contextSpec)`** — resolves the signed-in caller's real AWS credentials (their Cognito Identity Pool *authenticated* role) for constructing plain AWS SDK v3 clients (e.g. to invoke a Lambda directly). Only valid inside `withAmplifyAuth`.
- **`amplifyRegion`** — the AWS region backing the project's Identity Pool, derived from `amplifyConfig.Auth.Cognito.identityPoolId`.
- **`amplifyOutputs`** — the raw parsed `amplify_outputs.json`, including the `custom` block written by `backend.addOutput(...)` (e.g. `custom.workspaceMembershipFunctionName`, `custom.stripeWebhookUrl`).

There is **no `getServerIamDataClient()`** — see [Authorization Model](#authorization-model) for why it was removed.

```typescript
// Authenticated route
import { withAmplifyAuth, getServerUserPoolDataClient } from '@mmshark/amplify-layer/server/utils/amplify'

export default defineEventHandler(async (event) => {
  return await withAmplifyAuth(event, async (contextSpec) => {
    const client = getServerUserPoolDataClient()
    const { data } = await client.models.UserProfile.get(contextSpec, { userId: event.context.user.userId })
    return { profile: data }
  })
})

// Public route (SubscriptionPlan only)
import { withAmplifyPublic, getServerPublicDataClient } from '@mmshark/amplify-layer/server/utils/amplify'

export default defineEventHandler(async () => {
  return await withAmplifyPublic(async (contextSpec) => {
    const client = getServerPublicDataClient()
    const { data } = await client.models.SubscriptionPlan.list(contextSpec, { filter: { isActive: { eq: true } } })
    return { plans: data }
  })
})
```

## Workspace Group Helpers (`server/utils/workspaceGroups.ts`)

Pure helpers for the group-per-workspace tenancy model, shared by the backend Lambdas and this layer:

- **`workspaceReaderGroup(workspaceId)`** → `` `ws:${workspaceId}:members` `` — every member (OWNER/ADMIN/MEMBER); grants read access to rows whose `readerGroups` field contains it.
- **`workspaceWriterGroup(workspaceId)`** → `` `ws:${workspaceId}:admins` `` — OWNER/ADMIN marker; grants **no** direct AppSync access (tenant tables are read-only for clients), kept as role metadata the privileged Lambdas use.
- **`workspaceGroupFields(workspaceId)`** → `{ readerGroups: [...], writerGroups: [...] }`, the exact shape every tenant record must be created with.

Cognito group membership is stamped into the user's access/ID token at issue time — group changes (creating/joining a workspace) are invisible until the next token refresh. See `layers/workspaces/composables/useWorkspaces.ts`'s `createWorkspace()` for the `fetchAuthSession({ forceRefresh: true })` pattern.

## Workspace Membership Invocation (`server/utils/workspaceMembership.ts`)

Shared helper for calling the backend's `workspace-membership` Lambda (`apps/backend/amplify/functions/workspace-membership/`), which owns every write to `Workspace`/`WorkspaceMember`/`WorkspaceInvitation`/`WorkspaceSubscription`. Lives in this layer (rather than in `workspaces`) so both the `workspaces` layer (workspace CRUD, invitations, membership) and the `billing` layer (subscription bootstrap) can use it without a dependency cycle.

- **`getSessionAccessToken(event)`** — reads the caller's Cognito access token off `event.context.session` (set by the workspaces layer's auth middleware); 401s if absent.
- **`readInvitationToken(event)`** — reads an invitation `token` from the query string or JSON body, whichever is present.
- **`invokeWorkspaceMembership<T>(contextSpec, accessToken, payload)`** — must run inside `withAmplifyAuth`. Resolves the caller's own authenticated Identity Pool credentials via `getAwsCredentials`, constructs an `@aws-sdk/client-lambda` `LambdaClient`, and invokes `amplifyOutputs.custom.workspaceMembershipFunctionName` with `{ ...payload, accessToken }`. The Lambda independently re-verifies the access token (Cognito `GetUser`) and re-checks OWNER/ADMIN rules — it never trusts the payload's claimed identity, so invoking it directly grants nothing beyond what the routes already expose. Throws a `createError` (mapping the Lambda's own `{ ok: false, statusCode, message }` failure shape, or a generic 500 on a Lambda-level error) so callers can just `await` it.

```typescript
import { withAmplifyAuth } from '@mmshark/amplify-layer/server/utils/amplify'
import { getSessionAccessToken, invokeWorkspaceMembership } from '@mmshark/amplify-layer/server/utils/workspaceMembership'

export default defineEventHandler(async (event) => {
  return await withAmplifyAuth(event, async (contextSpec) => {
    const accessToken = getSessionAccessToken(event)
    const body = await readBody(event)
    return await invokeWorkspaceMembership(contextSpec, accessToken, {
      action: 'createWorkspace',
      name: body.name
    })
  })
})
```

## Logger (`utils/logger.ts`)

```typescript
import { createLogger } from '@mmshark/amplify-layer/utils/logger'

const logger = createLogger('workspaces:invite')
logger.debug('payload', body)   // dev-only (import.meta.dev)
logger.info('invited', email)   // dev-only
logger.warn('retrying...')      // always logs
logger.error('failed', err)     // always logs
```

`createLogger(scope)` returns `{ debug, info, warn, error }`, each prefixed `[scope]`. `debug`/`info` are silenced outside development; `warn`/`error` always log. Prefer this over ad hoc `console.*` calls in new server-side code (Nitro routes, Amplify Lambda handlers) so output stays consistent and filterable.

## Usage Examples

### Client-side (browser/SSR component)

```vue
<script setup lang="ts">
const { $Amplify } = useNuxtApp()

const { data: plans } = await $Amplify.GraphQL.client.models.SubscriptionPlan.list({
  filter: { isActive: { eq: true } }
})
</script>
```

### Storage

```typescript
const { $Amplify } = useNuxtApp()

await $Amplify.Storage.uploadData({
  path: `uploads/${file.name}`,
  data: file,
  options: { contentType: file.type }
})

const { url } = await $Amplify.Storage.getUrl({ path: `uploads/${file.name}`, options: { expiresIn: 3600 } })
```

### Nitro route reading owner-scoped data

```typescript
// server/api/profile.get.ts
import { withAmplifyAuth, getServerUserPoolDataClient } from '@mmshark/amplify-layer/server/utils/amplify'

export default defineEventHandler(async (event) => {
  return await withAmplifyAuth(event, async (contextSpec) => {
    const client = getServerUserPoolDataClient()
    const { data } = await client.models.UserProfile.get(contextSpec, { userId: event.context.user.userId })
    return { profile: data }
  })
})
```

## Troubleshooting

**"Not Authorized to access..."**
Wrong `authMode` for the model/operation, or the caller isn't in the Cognito group the record's `readerGroups`/`writerGroups` requires. Check `apps/backend/amplify/data/resource.ts` for the model's actual authorization rules, and confirm the caller's session token has the expected `cognito:groups` claim (remember: group changes need a token refresh to take effect).

**"No AWS credentials available for the current Amplify server context"**
`getAwsCredentials()`/`invokeWorkspaceMembership()` was called outside `withAmplifyAuth`, or the request has no valid session. `withAmplifyPublic` deliberately carries no credentials provider.

**Amplify not configured / `amplify_outputs.json` missing**
Run `pnpm amplify:sandbox:generate-outputs` (sandbox) after `pnpm backend:sandbox:init`, or `pnpm amplify:generate-outputs` for a deployed branch.

**`custom.workspaceMembershipFunctionName` / `custom.stripeWebhookUrl` missing from `amplify_outputs.json`**
Redeploy the backend and regenerate outputs — these are written by `backend.addOutput(...)` in `apps/backend/amplify/backend.ts` and only appear after a deploy that includes those functions.
